# Cozy Cafe — Website

Marketing website for Cozy Cafe, a Turkish & British café on Ferry Road, Leith, Edinburgh. Built with Astro, Tailwind CSS v4 and TypeScript, shipped as a fully static site with near-zero client-side JavaScript.

## Stack

- **[Astro](https://astro.build)** — static output, islands architecture
- **[Tailwind CSS v4](https://tailwindcss.com)** — CSS-first config (design tokens live in `src/styles/global.css`, not a `tailwind.config.js` — see [Design tokens](#design-tokens) below)
- **TypeScript** (strict) throughout, including data files and client scripts
- **[Fontsource](https://fontsource.org)** variable fonts, self-hosted (no Google Fonts, no render-blocking requests)
- **[@astrojs/sitemap](https://docs.astro.build/en/guides/integrations-guide/sitemap/)** for `sitemap-index.xml`; `robots.txt` is generated by `src/pages/robots.txt.ts`
- A handful of tiny vanilla-TS scripts for the only interactive pieces: mobile nav, sticky order bar, scroll-reveal, the live open/closed pill, menu scroll-spy and cookie consent. No UI framework, no jQuery, no component kit.

## Project structure

```text
src/
├── components/       Reusable, typed Astro components (Header, Footer, MenuSection, DealCard, …)
├── data/
│   ├── site.ts       Single source of truth for every business fact (hours, address, prices, links…)
│   └── menu.ts       The full menu, typed, rendered on both / and /menu
├── lib/
│   ├── hours.ts       Open/closed calculation in Europe/London time
│   └── schema.ts       JSON-LD builders (Restaurant, Menu, BreadcrumbList)
├── layouts/
│   └── Layout.astro   Shared <head>, header, footer, sticky bar, cookie banner
├── pages/              One file per route (/, /menu, /about, /visit, /deals, legal stubs, robots.txt)
├── scripts/            Small vanilla-TS files loaded as <script src="..."> — no bundler magic needed
└── styles/
    └── global.css      Tailwind import + @theme design tokens + base styles
public/                 Static assets: favicon, og-image.jpg
scripts/
└── generate-og-image.mjs  One-off script that renders the placeholder OG image (see TODO.md)
```

## Editing content

**Everything factual about the business lives in `src/data/site.ts`** — name, tagline, rating, address, phone, geo-coordinates, social links, ordering links, opening hours, and current offers. Nothing else in the codebase should hardcode a business fact; every page/component imports from this file. Update it once and every page (including the JSON-LD structured data and the live open/closed pill) updates automatically.

**The menu lives in `src/data/menu.ts`** as a typed array of categories, each with items that have a `price` (single) or `prices` (an array of `{ label, price }` for items with options like "plain / with chips"). Add `badges: ['Popular']` / `['Vegetarian']` / `['Spicy']` / `['Deal']` to flag items — `Badge.astro` renders them consistently everywhere. The category `id` is used both as the anchor on `/menu#id` and as the scroll-spy target, so keep it a URL-safe slug.

**Opening hours**: edit the `hours` array in `site.ts` (24h `"HH:MM"` strings, `day` is 0=Sunday…6=Saturday). The live "Open now / Closed" pill (`OpenStatusPill.astro` + `src/lib/hours.ts` + `src/scripts/open-status.ts`) recalculates from this array in the `Europe/London` timezone — no separate place to update.

**Current offers/deals**: edit the `offers` array in `site.ts`. These power both the `/deals` page and the deals band on the homepage.

## Swapping images

Real photography is not yet in the project — see `TODO.md` for the full shot list. Every image slot uses `FoodImage.astro`, which renders a tasteful, correctly-sized placeholder (so there's no layout shift) when no `src` is passed. To drop in a real photo:

1. Add the optimised image file to `public/images/` (export as `.webp` or `.jpg`, appropriately sized — e.g. ~1200px wide for hero/dish photos).
2. Pass `src="/images/your-file.jpg"` to the relevant `<FoodImage>` call (in `Hero.astro`, `DishCard.astro`, `about.astro`, etc.) along with a descriptive `alt`.
3. Keep the same `aspect` prop so the layout doesn't shift.

The OG share image (`public/og-image.jpg`) is a generated placeholder (`npm run generate-og-image`, uses `sharp`). Replace the file directly with real brand photography sized 1200×630 before launch.

## Design tokens

Tailwind v4 uses a **CSS-first config** — there is no `tailwind.config.js`. All design tokens (colours, fluid type scale, radii, shadows, spacing) are declared in the `@theme` block at the top of `src/styles/global.css`. Brand colours (`espresso`, `clay`, `paprika`, `cream`, `oat`, `sage`, `honey`, `ink-muted`) are exactly as specified by the brand. Two additional tokens, `clay-ink` and `sage-ink`, are deepened variants used only for text/links on light backgrounds — the literal brand `clay`, `sage` and `paprika` hues don't reach 4.5:1 contrast at body-text sizes, so interactive text uses the `-ink` variants instead while buttons/badges/icons keep the true brand hues.

## Enabling analytics

Analytics is off by default and gated behind cookie consent. To turn it on:

1. Set `analytics.enabled = true` in `src/data/site.ts`.
2. Choose `provider: 'plausible'` (set `plausibleDomain`) or `provider: 'ga4'` (set `ga4Id`).
3. That's it — `Analytics.astro` renders the script tag inert (`type="text/plain"`) and `src/scripts/cookie-consent.ts` only activates it after a visitor clicks "Accept" on the cookie banner (or immediately, if they already have). No tracking ID is hardcoded in this repo; you must supply a real one.

## Accessibility & performance

Built to hit Lighthouse mobile Performance ≥ 95 and Accessibility/Best Practices/SEO = 100, WCAG 2.2 AA, and CLS < 0.05. Verified against the production build with Lighthouse (mobile, simulated throttling) across every route. Notes:

- All colour combinations used for text (not just brand-colour buttons/badges/icons) were checked against WCAG's 4.5:1 threshold, including hover states — see the `-ink` token note above.
- Every interactive element has visible `:focus-visible`, `:hover` and `:active` states.
- All motion (scroll reveals, the sticky bar's hide/show, transitions) is disabled under `prefers-reduced-motion: reduce`.
- Images use `FoodImage.astro`, which enforces an `aspect-ratio` box whether or not a real photo is present, so there's no layout shift when photography is added later.
- Custom CSS in `global.css` that isn't a Tailwind utility (the open-status pill, menu chip nav) is wrapped in `@layer components`, so Tailwind's own utilities (e.g. `hidden`, responsive variants) always win the cascade as expected.

## Deploy

A `netlify.toml` is included with sensible cache headers (long-lived immutable caching for hashed `_astro/*` assets) and security headers (CSP in `Content-Security-Policy-Report-Only` to start, `X-Content-Type-Options`, `Referrer-Policy`, `Strict-Transport-Security`, `X-Frame-Options`). Adjust the CSP once you've watched the reports and are ready to enforce it (rename the header to `Content-Security-Policy`).

For Vercel or Cloudflare Pages, the build is a standard static Astro build (`npm run build` → `dist/`); port the header rules from `netlify.toml` into that platform's config format (`vercel.json` headers, or a `_headers` file for Cloudflare Pages).
